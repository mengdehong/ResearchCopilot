## 链路 1：发消息 → Agent 执行 → 结果显示

| 步骤                         | 当前实现                                                                                                                                            | 问题                                                                                                                                                                                                                                     | 严重度 | 修复方案                                                                                   |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ | ------------------------------------------------------------------------------------------ |
| 用户输入 → addMessage        | `WorkbenchPage.handleSendMessage` 创建本地 Message 对象                                                                                             | ✅ 正常                                                                                                                                                                                                                                   | —      | —                                                                                          |
| 自动创建 Thread              | `useCreateThread` → POST `/agent/threads`                                                                                                           | ✅ 正常，`justCreatedRef` 防止 reset                                                                                                                                                                                                      | —      | —                                                                                          |
| createRun → trigger_run      | `agent_service.trigger_run` 存 RunSnapshot + 启动图                                                                                                 | ⚠️ `editor_content` / `attachment_ids` 虽然 RunSnapshot 模型有字段，但 `trigger_run` **从未传入**，始终为 NULL                                                                                                                            | 中     | `trigger_run` 接收并保存 `editor_content` 和 `attachment_ids`                              |
| SSE event: token             | `event_translator` 中 `on_chat_model_stream` → token，但注释明确说"当前所有 LLM 都用 `with_structured_output().invoke()`，不产生面向用户的流式文本" | ⚠️ **token 事件永远不会触发**。前端 `generatedContent` 累积逻辑和 EditorTab 的流式插入逻辑是**死代码**                                                                                                                                    | 中     | 当增加面向用户的流式 LLM 调用时再启用；当前应在文档中标注                                  |
| SSE event: node_start/end    | `translate_to_run_event` 过滤有 `langgraph_node` 的事件                                                                                             | ⚠️ 同名节点多次执行时 `cotTree` 中 `node_end` 用 `n.name === endNodeName && n.status === 'running'` 匹配，仅更新**第一个**同名 running 节点，后续同名节点无法正确结束                                                                     | 低     | 改用 `node_id` 匹配而非 `node_name`                                                        |
| SSE event: assistant_message | 流结束后由 `event_generator` 注入，内容来自 `last_ai_content`（supervisor on_chain_end 的 AIMessage）                                               | ⚠️ 仅从 `supervisor` 节点提取 AI 内容。**其他 workflow 子图中 LLM 的自然语言输出被忽略**                                                                                                                                                  | 中     | 扩展 `last_ai_content` 提取范围，或在 workflow 节点 end 时也收集                           |
| SSE event: content_block     | WF 子图 node_end 时从 `raw_output.artifacts` 提取 → `render_artifacts` → 注入                                                                       | ⚠️ `raw_event.data.output` 在 `translate_to_run_event` 调用后已被覆盖为精简 `data`，但 content_block 逻辑用的是**原始** `raw_event`，实际可行                                                                                             | —      | —                                                                                          |
| SSE event: run_end           | `finally` 块始终发送                                                                                                                                | ⚠️ `run_end` 事件中 `data.run_id` 是字符串，前端 `handleSSEEvent` 的 `run_end` case **不使用 data**，直接 `set({ isStreaming: false })`。功能正常但数据被丢弃                                                                             | 低     | 前端记录 run_id 以供调试                                                                   |
| assistant_response 持久化    | `finally` 中保存到 `RunSnapshot.assistant_response`                                                                                                 | ⚠️ 使用 `request.app.state.session_factory` 新建 session（因为原 session 可能已关闭），但 **interrupted run 也会执行 finally**，此时 `last_ai_content` 为 None 且 `completed_nodes` 可能非空，会写入"已完成工作流"文本即使 run 未真正完成 | 中     | `was_interrupted` 为 true 时跳过 `assistant_response` 保存，或标记 status 为 "interrupted" |
| RunSnapshot.status 更新      | `finally` 中硬编码 `status = "completed"`                                                                                                           | 🔴 **错误/取消/中断的 run 也被标记为 completed**。无 "failed" / "interrupted" / "cancelled" 状态区分                                                                                                                                      | 高     | 根据 `was_interrupted`, 异常类型和 cancel 状态设置正确的 status                            |
| Thread.status 更新           | **从未更新**。创建时 `server_default="creating"`，`agent_service.create_thread` 设为 `"idle"`                                                       | 🔴 Thread 创建后状态**始终停留在 "idle"**。run 开始/结束/中断都不更新 Thread.status                                                                                                                                                       | 高     | 在 `trigger_run`/resume/cancel 及 SSE 结束时更新 Thread.status                             |

## 链路 2：切换 Thread → 状态恢复

| 步骤                           | 当前实现                                                                                       | 问题                                                                                                                                                              | 严重度 | 修复方案                                                               |
| ------------------------------ | ---------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ | ---------------------------------------------------------------------- |
| 侧边栏点击 → URL ?thread= 变化 | `WorkbenchPage` 通过 `useSearchParams` 读取                                                    | ✅ 正常                                                                                                                                                            | —      | —                                                                      |
| reset store                    | `useEffect` 检测 `threadId !== prevThreadRef.current` → `reset()`                              | ✅ 正确清空所有状态                                                                                                                                                | —      | —                                                                      |
| 加载历史消息                   | `useMessages(threadId)` → GET `/agent/threads/{id}/messages`                                   | ⚠️ `loadMessages` 条件为 `currentMessages.length === 0`。如果 reset 和 useQuery 的竞态导致 messages 未被清空，历史消息**不会加载**                                 | 中     | 改为 `historyMessages` 变化时直接 set，不检查 `currentMessages.length` |
| 消息序列化                     | GET messages 从 `RunSnapshot` 重建：每个 snap → user + optional assistant                      | ⚠️ **system 消息（错误提示）不被持久化**。error SSE event 产生的 system 消息是内存态，切走即丢                                                                     | 低     | 可接受——错误消息为临时提示                                             |
| CoT 树恢复                     | **不恢复**。`cotTree` 仅内存态                                                                 | ⚠️ 切走再切回，**CoT 树消失**。对于已完成的 run 无法回看思考过程                                                                                                   | 中     | 将 CoT tree 持久化到 RunSnapshot 或单独表                              |
| contentBlock 恢复              | **不恢复**。`contentBlock` 仅内存态                                                            | ⚠️ contentBlock 在 EditorTab 中 `setContent` 后被清空（`setState({ contentBlock: null })`），但如果用户**未打开 editor tab** 就切走，contentBlock 内容**永久丢失** | 中     | contentBlock 应存入 DB（RunSnapshot 或专用列），切回时从 DB 恢复       |
| editor draft 恢复              | `useDraft(threadId)` → GET `/editor/draft/{threadId}`                                          | ✅ 正确。EditorTab 的 `useEffect` 在 `draft?.content` 变化时 `setContent`                                                                                          | —      | —                                                                      |
| 进行中 run 的 SSE 重连         | **不支持**。`activeRunId` 是 local state，切走后清空。切回时没有检查 thread 是否有 running run | 🔴 如果 thread 有正在进行的 run，切走再切回**无法恢复 SSE 连接**，run 的后续事件全部丢失                                                                           | 高     | 切回时查询 `GET /runs`，如有 running run 则恢复 SSE 连接               |
| activePdf / sandboxResult      | **不恢复**。仅内存态                                                                           | ⚠️ 切走后 PDF 高亮位置和沙盒结果丢失                                                                                                                               | 中     | sandboxResult 应存入 RunSnapshot；activePdf 可后续从 RAG 重新定位      |

## 链路 3：HITL 中断 → 用户交互 → 恢复执行

| 步骤                            | 当前实现                                                                                                       | 问题                                                                                                                                                                                                                                                                                            | 严重度                                                                                                          | 修复方案                                                                               |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| interrupt 事件生成              | `LangGraphRunner._execute` 检测 `state_snapshot.next` → 提取 interrupt payload → push `__interrupt__` 到 queue | ✅ 正常                                                                                                                                                                                                                                                                                          | —                                                                                                               | —                                                                                      |
| interrupt 事件翻译              | `translate_to_run_event`: action, run_id, thread_id, title, description, papers(从 candidates)                 | ⚠️ **仅提取 6 个字段**。对于 `confirm_execute` 类型，`code` 字段在翻译时**被丢弃**。对于 `confirm_finalize`，`content` 字段也**被丢弃**                                                                                                                                                          | 🔴 高                                                                                                            | 翻译时保留完整 payload 或按 action type 提取对应字段（code, content 等）               |
| 前端 handleSSEEvent interrupt   | 构建 `InterruptData`，`payload` 设为 `data as Record<string, unknown>`                                         | ⚠️ 由于后端翻译时已丢弃字段，前端 `interrupt.payload.code` 和 `interrupt.payload.content` 始终为 undefined                                                                                                                                                                                       | 🔴 高                                                                                                            | 配合后端修复，确保 payload 完整                                                        |
| HITLCard 显示                   | `ConfirmExecuteCard` 读取 `interrupt.payload.code`，`ConfirmFinalizeCard` 读取 `interrupt.payload.content`     | 🔴 由于上述问题，**代码预览和内容预览始终为空**。用户看到的是空白或"无代码"                                                                                                                                                                                                                      | 🔴 高                                                                                                            | 后端 interrupt 翻译修复后，前端自动恢复                                                |
| PaperSelectOverlay              | 读取 `interrupt.payload.papers ?? interrupt.payload.candidates`                                                | ⚠️ 后端翻译时将 `candidates` 映射为 `papers` key，前端有 fallback。**基本可用**                                                                                                                                                                                                                  | 低                                                                                                              | 统一命名为 `papers`                                                                    |
| resume 请求                     | POST `/resume` → `InterruptResponse { action, selected_ids?, payload?, feedback? }`                            | ⚠️ 前端 `handleResumeInterrupt` 的 `body` 构建：`{ action, payload }`。但 `InterruptResponse` schema 要求 `selected_ids` 为独立字段。PaperSelectOverlay 传的是 `payload: { selected_ids: [...] }`，后端 `resume_run` 中 `body.selected_ids` 为 None，而是从 `body.payload` 中用 `.update()` 合并 | ⚠️ 功能偶然正确（因为 `resume_payload.update(body.payload)` 把 `selected_ids` 注入了），但**绕过了 schema 验证** | 低                                                                                     | 前端改为直接传 `selected_ids` 字段 |
| resume 后新 SSE                 | resume 返回 `new_run_id` → 前端 `setActiveRunId(result.run_id)` → `useSSE` 重新连接                            | ✅ 正常                                                                                                                                                                                                                                                                                          | —                                                                                                               | —                                                                                      |
| 页面刷新后 interrupt 恢复       | **不支持**。interrupt 数据仅存于 Zustand 内存态                                                                | 🔴 **页面刷新后 HITL 卡片消失，interrupt 永久丢失**。用户只能取消 run 或重新开始                                                                                                                                                                                                                 | 高                                                                                                              | RunSnapshot 增加 interrupt 状态列。GET messages 或 GET thread 时返回 pending interrupt |
| interrupt 时 RunSnapshot.status | resume_run 创建新 RunSnapshot(`status="running"`)，但**原 run 的 snapshot 状态不更新**                         | ⚠️ 被中断的 snap 状态从 SSE finally 中被设为 "completed"（见链路 1），实际应为 "interrupted"                                                                                                                                                                                                     | 中                                                                                                              | 中断时更新原 snap 为 "interrupted"                                                     |

## 链路 4：Editor/Canvas 内容持久化

| 步骤                          | 当前实现                                                                                                             | 问题                                                                                                                                                                       | 严重度 | 修复方案                                                              |
| ----------------------------- | -------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ | --------------------------------------------------------------------- |
| content_block SSE → EditorTab | contentBlock 触发 `marked.parse` → `editor.commands.setContent(html)` → 清空 contentBlock                            | ⚠️ `setContent` 是**替换**操作。如果 editor 中有用户手写内容，会被**完全覆盖**                                                                                              | 中     | 改为 `insertContent` 追加到末尾，或提示用户确认                       |
| 多个 WF 的 content_block      | 每个 WF 子图 node_end 时都可能注入 content_block                                                                     | ⚠️ 后一个 content_block **直接替换**前一个（`set({ contentBlock: ... })`），如果前端未及时消费（EditorTab useEffect 尚未执行），**前一个 content_block 数据丢失**           | 中     | 改为 content_block 队列（数组 push），EditorTab 逐个消费              |
| content_block 持久化          | **不持久化**。仅 Zustand 内存态 + EditorTab 即时写入 editor                                                          | ⚠️ content_block 本身不入 DB。但如果被写入 editor → 触发 debounce save → `useSaveDraft` → PUT `/editor/draft`，则**间接持久化**。条件：EditorTab 必须已挂载且 tab 为 editor | 中     | 直接将 content_block 存入 DB 作为 run artifacts                       |
| useDraft threadId 依赖        | `useDraft(threadId)` + `useSaveDraft({ threadId, content })`                                                         | ✅ threadId 来自 props 稳定传递                                                                                                                                             | —      | —                                                                     |
| editor draft debounce 保存    | `editor.on('update', debouncedSave)` → 2s 后 `handleSave`                                                            | ⚠️ 如果用户在 2s 内切走 thread，`saveTimerRef` 被 cleanup 清除，**最后一次编辑可能丢失**                                                                                    | 中     | cleanup 时执行一次 `handleSave`（flush pending save）                 |
| Draft 与 contentBlock 的时序  | `useEffect([draft, editor])` → `setContent(draft.content)`；`useEffect([editor, contentBlock])` → `setContent(html)` | ⚠️ draft 加载和 contentBlock 写入可能**竞争**。如果 draft 在 contentBlock 之后加载，会覆盖 contentBlock。反之亦然                                                           | 中     | 仅在 threadId 变化时加载 draft；contentBlock 始终 append 而非 replace |
| generatedContent 流式插入     | `useEffect([editor, generatedContent])` → `insertContent(newContent)`                                                | ⚠️ 当前 token 事件**永远不触发**（见链路 1），此逻辑为死代码                                                                                                                | 低     | 标注为未来启用                                                        |

## 链路 5：文档上传 → 解析 → Agent 引用

| 步骤                          | 当前实现                                                                                   | 问题                                                                                                                                                                                        | 严重度 | 修复方案                                                                        |
| ----------------------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ | ------------------------------------------------------------------------------- |
| 前端上传流程                  | `DocumentsPage`: initiate_upload → PUT upload_url → confirm_upload                         | ✅ 基本可用                                                                                                                                                                                  | —      | —                                                                               |
| 上传错误处理                  | `uploadFile` 的 catch 块为空（`// Error is visible via React Query error state`）          | ⚠️ React Query 的 error state **未被渲染**。`initiateUpload` mutation 的 error 不会显示给用户                                                                                                | 中     | 添加 toast 通知或在 UI 中显示 error state                                       |
| confirm_upload → Celery       | `document_service.confirm_upload` → `ingest_document.delay(doc_id, thread_id, run_id)`     | ⚠️ 前端 `useConfirmUpload` **不传 thread_id / run_id**。仅传 `document_id`。从 DocumentsPage 上传的文档永远不会有 agent context                                                              | 低     | DocumentsPage 上传属于手动管理，可接受；Agent 触发的下载通过 `/from-arxiv` 传递 |
| Celery ingestion 管道         | `ingest_document` → `run_parse_pipeline` → `_persist_classified_content`                   | ✅ 四阶段管道 + 持久化                                                                                                                                                                       | —      | —                                                                               |
| ingestion webhook 回调        | POST `/agent/threads/{thread_id}/runs/{run_id}/resume` with `action: "ingestion_complete"` | ⚠️ resume 端点**要求鉴权**（`get_current_user`），webhook 用 system JWT。但 `get_current_user` 期望 `sub` 是用户 ID 并从 DB 查 User。system token 的 `sub="system"` **可能查不到 User 记录** | 🔴 高   | 为内部系统调用添加 bypass 或使用 `get_current_user_sse` 的系统令牌支持          |
| ingestion 完成后 Agent 可检索 | Agent 通过 RAG 工具查询 pgvector                                                           | ⚠️ **没有主动通知 Agent 文档已可用的机制**（webhook 可能失败，见上）。Agent 的 `wait_for_ingestion` interrupt 可能**永远不被 resume**                                                        | 高     | 修复 webhook 鉴权；增加重试和超时检测                                           |
| PDF 高亮数据持久化            | `activePdf` 仅内存态。`pdf_highlight` SSE 事件将 bbox/page 存入 Zustand                    | ⚠️ **不持久化**。切走/刷新后高亮位置丢失                                                                                                                                                     | 中     | 将 highlights 存入 RunSnapshot 或 Document 关联表                               |
| PDF 高亮跨 thread             | `activePdf` 绑定 Zustand 全局，切换 thread 时 `reset()` 清空                               | ⚠️ 高亮不跨 thread，也不跨 run。每个 run 的 RAG 溯源结果是即时的                                                                                                                             | 低     | 如需持久化可在 DB 中关联 document_id + run_id                                   |
