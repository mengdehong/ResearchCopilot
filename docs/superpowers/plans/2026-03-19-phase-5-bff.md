# Phase 5: FastAPI BFF — 细化实施计划

> BFF 层职责：鉴权、租户隔离、Agent 状态桥接、SSE 事件翻译、文件代理。不运行 Agent 逻辑。

## 现状盘点

| 已有组件                                | 文件                        | 状态               |
| --------------------------------------- | --------------------------- | ------------------ |
| FastAPI 入口                            | `backend/main.py`           | ✅ 含 `/health`     |
| ORM Models（全量）                      | `backend/models/` 17 个文件 | ✅ 完整             |
| Config + Database + Logger + Exceptions | `backend/core/`             | ✅ 完整             |
| API 目录骨架                            | `backend/api/`              | ⬜ 仅 `__init__.py` |
| Repository 目录                         | `backend/repositories/`     | ⬜ 仅 `__init__.py` |
| Services 目录                           | `backend/services/`         | ✅ Agent 侧 4 模块  |
| Workers 目录                            | `backend/workers/`          | ⬜ 仅 `__init__.py` |

> [!NOTE]
> Phase 5 与 Phase 4 **没有依赖关系**，可并行开发。Phase 5 仅依赖 Phase 1（ORM）和 Phase 2（LLM Gateway 用于 quota 统计）。

## 设计决策

### 1. 认证方案
固定 JWT secret mock 认证（MVP）。后续按需替换为第三方 Auth SDK。

### 2. LangGraph Server 通信
定义 `LangGraphClient` 接口 + mock 实现。Phase 8 集成时切换为真实 `langgraph-sdk` HTTP 连接。

### 3. 对象存储
本地文件系统 + `StorageClient` 接口抽象。Phase 8 切换 MinIO。

### 4. Service 层目录
统一平铺在 `backend/services/`。Agent 侧和 BFF 侧通过文件命名区分。

## Task 4: Repository 层

### 文件产出

| 文件                                        | 说明                               |
| ------------------------------------------- | ---------------------------------- |
| `backend/repositories/base.py`              | `BaseRepository[T]` 泛型 CRUD 基类 |
| `backend/repositories/user_repo.py`         | 用户 CRUD                          |
| `backend/repositories/workspace_repo.py`    | Workspace + 成员查询               |
| `backend/repositories/document_repo.py`     | 文档 CRUD + 状态流转               |
| `backend/repositories/editor_repo.py`       | 草稿 UPSERT                        |
| `backend/repositories/run_snapshot_repo.py` | Run 快照 + 状态查询                |
| `backend/repositories/quota_repo.py`        | Token 消耗记录 + 统计              |
| `tests/integration/test_repositories.py`    | 集成测试（需 PostgreSQL）          |

### 核心接口

```python
class BaseRepository(Generic[T]):
    """泛型 CRUD 基类。不管理事务，接收 AsyncSession 作为参数。"""
    async def get_by_id(self, id: UUID) -> T | None: ...
    async def create(self, **kwargs) -> T: ...
    async def update(self, id: UUID, **kwargs) -> T: ...
    async def soft_delete(self, id: UUID) -> None: ...

class DocumentRepo(BaseRepository[Document]):
    async def update_status(self, doc_id: UUID, new_status: str) -> Document: ...
    async def count_by_status(self, workspace_id: UUID) -> DocStatusCounts: ...
    async def list_by_workspace(self, workspace_id: UUID, *, status_filter: str | None = None) -> list[Document]: ...
```

### 关键实现

- `update_status` 内含状态流转守卫函数，非法转换抛 `InvalidStateTransitionError`
- 所有 repo 不管理事务生命周期，由 `get_db()` 的 context manager 控制

### 测试要点

- 集成测试需要真实 PostgreSQL（`docker compose` 启动测试数据库）
- 验证 `update_status` 状态守卫（uploading→pending ✅, completed→parsing ❌）
